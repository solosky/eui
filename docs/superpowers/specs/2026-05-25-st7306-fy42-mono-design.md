# ST7306 / FY42-Mono Display Driver — Design Spec

**Date:** 2026-05-25  
**Status:** Approved  
**Scope:** Add ST7306 display driver for FY42-Mono (YDP420H001-V3), 300×400 2bpp, plus ESP-IDF CMake integration.

## 1. Background

FY42-Mono is a 4.2" 300×400 reflective TFT with an ST7306 controller (SPI interface). The project already has 2bpp canvas/font support (commits `8b9ed1e`–`5227b5b`). A reference implementation exists at the project root as `st7306.h` (Retro-Go style, FreeRTOS SPI task architecture). This spec adapts that reference into a proper EUI display driver following the project's HAL abstraction pattern.

## 2. Driver Architecture

### Files

| File | Purpose |
|------|---------|
| `include/eui/driver/eui_drv_st7306.h` | Public header: config struct + create/destroy prototypes |
| `src/driver/eui_drv_st7306.c` | Implementation: SPI transport, pixel packing, init sequence |

### Data Structures

```c
// Public config (user-facing)
typedef struct {
    eui_hal_spi_t spi;
    uint16_t      width;   // 300
    uint16_t      height;  // 400
} eui_drv_st7306_config_t;

// Internal driver state (private to .c)
typedef struct {
    eui_display_drv_t base;
    eui_hal_spi_t     spi;
    uint16_t          width;
    uint16_t          height;
    uint8_t          *framebuffer;  // 150×200 = 30000 bytes
} st7306_t;
```

### Inheritance

Follows the standard EUI pattern: `st7306_t.base` is the first member. `container_of` not needed — the `user_data` field of `eui_display_drv_t` points to `st7306_t`, and all vtable callbacks receive `user_data` as the `void *ud` parameter.

### Buffer Mode

`EUI_BUFFER_FULL` — 30KB native-format frame buffer allocated via `eui_malloc()`. The buffer mirrors the ST7306 GRAM layout exactly: 150 bytes per row × 200 rows, 2bpp per pixel.

### Pixel Packing (ST7306 Native Format)

Each byte represents a 2×2 block of 4 physical pixels, 2 bits each:

```
Bit:  [7] [6] [5] [4] [3] [2] [1] [0]
      P0H P1H P0L P1L P2H P3H P2L P3L

  P0 = top-left     (bit1=7, bit0=5)
  P1 = bottom-left  (bit1=6, bit0=4)
  P2 = top-right    (bit1=3, bit0=1)
  P3 = bottom-right (bit1=2, bit0=0)
```

Logical coordinate (x, y) → buffer offset:

- `bx = x / 2`, `by = y / 2`
- `byte_idx = by * 150 + bx`
- `sx = x % 2`, `sy = y % 2` (sub-pixel position within 2×2 block)
- bit1 position = `7 - (sx * 4 + sy)`
- bit0 position = `7 - (sx * 4 + 2 + sy)`

### CASET/RASET Write Window (for flush)

50 column addresses (5–54), each receiving 3 buffer bytes = 6 logical pixels.  
200 row addresses (0–199), each = 2 logical rows.

Flush rect mapping:

- `gram_col_start = rect.x / 6`
- `gram_col_end   = (rect.x + rect.w - 1) / 6`
- `gram_row_start = rect.y / 2`
- `gram_row_end   = (rect.y + rect.h - 1) / 2`
- CASET = `(5 + gram_col_start)` .. `(5 + gram_col_end)`
- RASET = `gram_row_start` .. `gram_row_end`

SPI data sent row-by-row: each row = `(gram_width * 3)` bytes.

### Init Sequence

Copied verbatim from the reference `st7306.h` (lines 406–467): NVM Load, Booster Enable, VCOM/Gate Voltage, OSC/Frame Rate, Gate Timing, Source EQ, Panel Setup, Sleep Out, MADCTL=0x48, COLMOD=0x11, Gamma Mode=0x20 (Mono), CASET 5–54, RASET 0–199, Display ON.

### Vtable Methods

| Method | Implementation |
|--------|---------------|
| `init` | Full hardware init sequence (~25 commands) |
| `deinit` | DISPOFF → SLPIN → free framebuffer |
| `draw_pixel` | Read-modify-write into framebuffer |
| `write_buffer` | Convert canvas pixels → native buffer → SPI flush via CASET/RASET+RAMWR |
| `set_power` | DISPON / DISPOFF |
| `set_invert` | INVON / INVOFF |
| `set_contrast` | No-op (reflective display, no backlight) |
| `fill_rect` | No-op (delegated to canvas) |

## 3. ESP-IDF Integration

### CMake Auto-Discovery

ESP-IDF path in root `CMakeLists.txt` already uses `file(GLOB EUI_DRIVER_SRCS src/driver/*.c)` with only `eui_drv_raylib.c` excluded. Placing `eui_drv_st7306.c` in `src/driver/` is sufficient — no CMake changes needed for ESP-IDF.

### Config Profile

`configs/esp32/st7306_spi_300x400_2bpp.cmake`:

```cmake
set(EUI_TARGET_PORT      "esp-idf")
set(EUI_COLOR_DEPTH      2)
set(EUI_DISPLAY_DRIVER   "st7306")
set(EUI_DISPLAY_WIDTH    300)
set(EUI_DISPLAY_HEIGHT   400)
set(EUI_BUFFER_MODE      "full")
set(EUI_MEM_POOL_SIZE    65536)
set(EUI_PIN_SPI_HOST     1)
set(EUI_PIN_MOSI         23)
set(EUI_PIN_SCLK         18)
set(EUI_PIN_CS           5)
set(EUI_PIN_DC           16)
set(EUI_PIN_RST          17)
set(EUI_INPUT_DRIVER     "buttons")
set(EUI_PIN_BTN_UP       34)
set(EUI_PIN_BTN_DOWN     35)
set(EUI_PIN_BTN_OK       32)
set(EUI_PIN_BTN_BACK     33)
```

### Kconfig

`configs/esp32/st7306_spi_300x400_2bpp.Kconfig` — menu entries for SPI pins, display dimensions. Structure mirrors existing `st7735_spi_240x240_16bpp.Kconfig`. No BL pin (reflective display).

### SPI HAL Factory

Add `eui_port_esp_idf_spi_create()` to `port/esp-idf/eui_port_esp_idf.c` (currently only has I2C factory). Implements `eui_hal_spi_t` (write_cmd, write_data, read_data, set_dc, set_cs, set_rst, delay_ms) using ESP-IDF `driver/spi_master.h`.

### Bootstrap Update

`port/esp-idf/eui_port_bootstrap.c` currently hardcodes SSD1306/I2C. Update to branch on `EUI_DISPLAY_DRIVER`:

```c
#if defined(EUI_DISPLAY_DRIVER) && EUI_DISPLAY_DRIVER == "st7306"
    // SPI + ST7306 path
#else
    // Existing I2C + SSD1306 path
#endif
```

## 4. Desktop CMake

Add `driver/eui_drv_st7306.c` to the explicit source list in `src/CMakeLists.txt`.

## 5. Memory Budget

| Item | Size |
|------|------|
| Framebuffer (150×200×1 byte) | 30,000 bytes |
| Driver struct + overhead | ~200 bytes |
| EUI core + widgets | ~4,000 bytes |
| **Recommended pool size** | **65,536 bytes** |

The framebuffer is allocated via `eui_malloc()` from the TLSF pool. Users must configure `EUI_MEM_POOL_SIZE >= 65536`.

## 6. Testing

- **Desktop**: Existing 400×300 2bpp launcher example (commit `0b6e29a`) confirms basic rendering
- **ESP32**: Requires physical hardware (FY42-Mono) for functional verification
- **Compile check**: Both `cmake --build build` (desktop) and `idf.py build` (ESP-IDF) must pass
